# Q-Zoom Examples

End-to-end templates for training and evaluating Q-Zoom models.

The Q-Zoom training pipeline has three (or four, for Qwen3-VL) stages.
Each stage trains a **different subset** of parameters — everything else
stays frozen. The selection is controlled by the `--enable_twig`,
`--roi_post_training`, and `--enable_high_res` flags consumed by
`train_qwen.py`.

| Stage | Trainable parts | Frozen parts | Flags |
|---|---|---|---|
| **Stage 1** | The `twig_T` twig layers only — the gating branch grafted on top of layer `twig_K`. | Base LLM (all layers, including the first `twig_K` ones), `lm_head`, vision encoder. | `enable_twig=True`, `roi_post_training=False`, `enable_high_res=False` |
| **Stage 2** | All LLM decoder layers + `lm_head`. This is the post-SFT pass that hardens the LLM under Q-Zoom gating. | TWIG branch (carried over from Stage 1), high-res gating network, vision encoder. | `enable_twig=True`, `roi_post_training=True` |
| **Stage 2.5** *(Qwen3-VL only)* | Same as Stage 1 — re-fit the `twig_T` twig layers on top of the Stage-2 LLM. | Stage-2 LLM, `lm_head`, vision encoder, high-res gating. | `enable_twig=True`, `roi_post_training=False`, `enable_high_res=False`, `is_2_5_stage=True` |
| **Stage 3** | High-resolution gating network only — `high_res_layers` + `high_res_head`. | Base LLM, `lm_head`, TWIG branch, vision encoder. | `enable_twig=True`, `roi_post_training=False`, `enable_high_res=True` |

> **Why is Stage 2.5 only for Qwen3-VL?** The Stage-2 LLM post-SFT pass
> shifts Qwen3-VL's intermediate hidden-state distribution enough that
> the TWIG branch trained in Stage 1 becomes mis-aligned with the new
> backbone features. We re-fit the twig layers for a few iterations on
> a small fixed-resolution pseudo-label set generated by the Stage-2
> LLM itself. Qwen2.5-VL backbones do **not** show meaningful drift
> after Stage 2, so they go straight from Stage 2 → Stage 3.

> The `--tune_mm_llm` / `--tune_mm_mlp` shell flags are **no-ops** once
> `--enable_twig=True` is set: the freeze/unfreeze logic in
> `train_qwen.py` overrides them with the table above.

Each example folder mirrors the same structure:

```
examples/<stage_name>/
├── README.md             # short walkthrough for this stage
├── 01_make_data.sh       # call standardized_pipeline to build training data
├── 02_train.sh           # launch torchrun training
├── 03_eval.sh            # evaluate the resulting checkpoint with lmms-eval
└── run_all.sh            # convenience driver for the steps above
```

All scripts use environment variables for paths so they are portable. The
defaults assume:

```
HOME_PATH=/path/to/home              # your home directory
CODE_ROOT=${HOME_PATH}/Q-Zoom        # this repository
DATA_ROOT=${HOME_PATH}/datasets      # raw + intermediate data
MODEL_PATH=Qwen/Qwen2.5-VL-7B-Instruct  # base HF model id (or local path)
NUM_GPUS=4
GPU_IDS=0,1,2,3
```

Override any of them on the command line, e.g.:

```bash
HOME_PATH=/scratch/me MODEL_PATH=/scratch/me/models/Qwen2.5-VL-7B-Instruct \
NUM_GPUS=8 GPU_IDS=0,1,2,3,4,5,6,7 \
bash examples/stage1_train_eval/run_all.sh
```

## Activate the right conda environment first

Q-Zoom requires **different `transformers` versions** for the two
backbones — they are not interchangeable. Activate the matching env
**before** running any of the example scripts:

| Backbone you are running | Conda env (created by `install.sh`) | `transformers` |
|---|---|---|
| Qwen2.5-VL-3B / 7B | `qzoom-q25` | `4.51.3` |
| Qwen3-VL-4B | `qzoom-q3` | `4.57.0` |

```bash
# Qwen2.5-VL experiments
conda activate qzoom-q25
bash examples/stage1_train_eval/run_all.sh

# Qwen3-VL experiments
conda activate qzoom-q3
MODEL_PATH=Qwen/Qwen3-VL-4B-Instruct TWIG_K=24 \
bash examples/stage1_train_eval/run_all.sh
```

If you mix them up, model loading will fail with an `ImportError` or a
config-mismatch error.

## Skip data generation: download our prepared training data

Generating Stage-1 pseudo labels and the Stage-2 / Stage-3 mixtures from
scratch is slow (hours on a multi-GPU box per stage). If you only want
to **train** Q-Zoom, you can download the prepared per-backbone files
from our Hugging Face dataset and point the training scripts straight at
them — see the per-stage README for the exact `huggingface-cli download`
recipe.

> **<https://huggingface.co/datasets/YuhengSSS/Q-Zoom-Training>**

| Skip step | Use prepared file (per backbone) |
|---|---|
| `stage1_train_eval/01_make_pseudo_labels.sh` | `qwen{2_5vl_3b,2_5vl_7b,3vl_4b}_pseudo_*_576res_185k.pkl` |
| `stage2_train_eval/01_build_stage2_data.sh`  | `qwen{2_5vl_3b,2_5vl_7b,3vl_4b}_stage2.jsonl` |
| `stage3_train_eval/01_build_stage3_data.sh`  | `qwen{2_5vl_3b,2_5vl_7b,3vl_4b}_stage3.pkl` |

Pre-built `stage{1,2,3}_universal_input.jsonl` files are also in the
repo if you want to skip the `build_universal_input.py` step.

## Eval-only quickstart

If you already have a Q-Zoom checkpoint (e.g. downloaded from HuggingFace),
skip straight to `examples/eval_only/`. It runs the standard Doc/OCR and
HR/Vision benchmarks against any checkpoint with TWIG enabled.
